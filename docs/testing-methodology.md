# 前端测试方法论 — 四层测试体系

**文档版本**: v1.1
**适用范围**: 本方法论适用于任何前端 SPA + 自建后端架构的 Web 项目

---

## 一、核心理念

测试的目标不是追求覆盖率数字，而是**确保软件行为符合预期，且变更不会破坏已有功能**。为此，我们将测试分为四个层次，每层有明确的测试对象、Mock 策略和价值目标。

```
┌──────────────────────────────────────────────────────────────────┐
│                        Mock 程度递增                              │
│                                                                   │
│  L1 单元测试     ────────  纯函数，零 Mock                         │
│       │                                                          │
│  L2 组件/页面测试  ───────  Mock 全部后端 API                      │
│       │                                                          │
│  L3 契约测试     ────────  真实后端路由，仅 Mock 外部服务           │
│       │                                                          │
│  L4 E2E 测试    ────────  能不 Mock 就不 Mock                     │
│                                                                   │
└──────────────────────────────────────────────────────────────────┘
```

---

## 二、第 1 层：单元测试（纯函数测试）

### 2.1 定义

测试前端中的**纯函数**和**工具函数**。不涉及任何后端 API 调用，不涉及 React 组件，不涉及 DOM。

### 2.2 测试对象

- `utils/` 目录下的工具函数（格式化、校验、计算、转换）
- `lib/` 目录下的纯逻辑模块
- 自定义的纯函数（日期处理、价格计算、参数拼接）
- 自定义错误类型（类构造与字段）

### 2.3 Mock 程度

**零 Mock**。输入 → 输出。不 mock 任何外部依赖。

### 2.4 通用价值

任何有 `utils.ts`、`helpers.ts`、`formatters.ts` 等工具函数文件的项目，都应写这层测试。这是成本最低、回报最高的测试层。

---

## 三、第 2 层：组件测试 / 页面测试（前端集成测试）

### 3.1 定义

测试 React 组件和页面的渲染与交互行为。Mock 全部后端 API。

### 3.2 细分

**组件测试**和**页面测试**虽然使用相同工具链，但粒度不同：

| 类型 | 测试范围 | 例子 |
|------|---------|------|
| 组件测试 | 单个可复用组件的渲染和交互 | ProductCard 传入不同 props→渲染不同内容 |
| 页面测试 | 完整页面的用户交互流程，Context + 路由 + 组件协同 | LoginPage：填表单→点发送→看到验证码输入框→登录跳转 |

### 3.3 测试对象

- **Context（状态管理）**: 状态机逻辑
- **自定义 Hooks**: 副作用和数据获取逻辑
- **可复用组件**: 不同 props 和状态的渲染结果
- **业务关键页面**: 登录、注册、结账等出错率高的流程

### 3.4 Mock 程度

Mock **全部后端 API**。用 `vi.mock()` 或 MSW 拦截所有网络请求。

### 3.5 通用价值

这层测试确保前端**本身就正确**，不依赖后端。如果你的后端挂了，这层测试仍然全绿——它能帮你快速定位问题在前端还是后端。

---

## 四、第 3 层：契约测试（前后端接口联调）

### 4.1 定义

验证前端和后端之间的 API 约定是否一致。包含两个子目标：

- **3a. 接口契约测试**：验证后端 API 返回的数据结构满足前端期望（字段名、类型、嵌套结构、错误码）
- **3b. API 覆盖度检查**：验证后端提供了前端业务所需的所有 API，不存在遗漏

### 4.2 3a 接口契约测试 — 做法

**直接复用后端已有的测试基础设施**，而不是在前端测试里再 mock 一遍网络。

具体做法：在测试文件中直接引入后端的 app 实例，调用 `app.request()` 模拟 HTTP 请求，断言返回的数据结构。

**Mock 程度**：只 Mock 后端依赖的外部服务，后端的路由处理逻辑完全真实执行。

### 4.3 3b API 覆盖度检查 — 做法

维护一份前端所需的完整 API 清单，逐条请求后端断言路由存在。这本身是一个**可执行的自动化测试**。

**价值**：如果某天后端重构时不小心删了一个路由，这个测试会立刻报红，在合并到主分支之前就发现问题。

### 4.4 Mock 程度

- 后端路由处理逻辑：**不 Mock**（真实执行）
- 后端依赖的外部服务：**Mock 掉**

### 4.5 通用价值

任何前后端分离项目都应该有这层测试。它是前后端之间的"编译器检查"，防止接口漂移导致的生产事故。

---

## 五、第 4 层：E2E 测试（端到端）

### 5.1 定义

从**用户视角**验证整个业务流程是否能跑通。使用真实浏览器（Playwright / Cypress）操作真实页面。

### 5.2 核心原则

> 能不 Mock 就不 Mock。遇到实在有阻塞的外部服务才 Mock，并且要用单独的测试点验证每个被 Mock 的服务是否真的可用。

### 5.3 阻塞点分析与处理策略

对于每个阻塞点，采用"E2E 里 Mock + 单独验证测试"的双保险策略。

### 5.4 框架选择

推荐 **Playwright**，理由：Vite 生态整合好、多浏览器支持、GitHub Actions 开箱可用。

### 5.5 通用价值

E2E 测试是"用户会骂什么我们就测什么"。它不测实现细节，只测业务流程。

---

## 六、视觉断言维度：UI/UX 测试

### 6.1 为什么不是第五层

UI/UX 测试天然地存在三种粒度：

| 子类型 | 测试对象 | 例子 | 对应的现有层级 |
|--------|---------|------|--------------|
| **视觉外观测试** | 组件/页面的渲染结果 | 颜色、字体、间距、遮挡、排版错乱、暗黑模式 | L2 组件/页面测试 |
| **交互体验测试** | 单个交互的反馈质量 | Loading 态是否出现、表单校验提示是否清晰、过渡动画是否流畅 | L2 页面测试 |
| **业务流程视觉测试** | 完整业务流程的视觉连续性 | 登录→浏览→下单，每一步的页面看起来都正常 | L4 E2E 测试 |

可以发现，UI/UX 测试所覆盖的**测试对象**与 L2、L4 完全重合。区别仅在于**断言方式不同**：

- L2/L4 的**逻辑断言**问的是：**"功能对不对？"** —— 数据正确、交互触发、流程走通
- UI/UX 的**视觉断言**问的是：**"看起来对不对？"** —— 布局合理、反馈及时、视觉正确

因此，UI/UX 测试不是一个新的独立层级，而是为 L2 和 L4 增加了一个**视觉断言维度**。L1（纯函数）和 L3（API 契约）不涉及 UI 渲染，不存在视觉测试的需求。

```
                    逻辑断言                        视觉断言
                (DOM/数据/流程校验)            (截图 + AI 视觉分析)

L1 单元测试         ✅ 输入→输出                    ❌ 不适用（无 UI）
L2 组件/页面        ✅ render() / text() / fireEvent  ✅ 截图 → AI 分析外观
L3 契约测试         ✅ 字段/类型/状态码              ❌ 不适用（无 UI）
L4 E2E             ✅ page.click() / expect()       ✅ 多页面截图 → AI 分析
```

> **核心思想**：UI/UX 测试是逻辑测试的"升维"，不是新加层。测试金字塔仍然是一座金字塔，但它现在有了色彩感知能力。

### 6.2 L2 视觉维度：组件与页面外观

**测什么**：
- 单个组件在不同 props/state 下的视觉正确性
- 页面在不同状态下的外观（空状态、加载态、错误态、正常态）
- 交互反馈的视觉质量

**为什么 L2 需要视觉测试**：
- L2 逻辑测试使用 jsdom，不真正渲染 CSS
- `expect(button).toBeInTheDocument()` 通过了，但真实浏览器中按钮可能被 CSS 覆盖
- CSS 变更、依赖升级导致的视觉漂移，jsdom 完全感知不到

**做法示例（使用 visual-test 框架）**：

```typescript
// visual-tests/components/ProductCard.visual.spec.tsx
import { test, expectVisual } from 'visual-test/ct'
import { ProductCard } from '../../src/components/ProductCard'

test('[视觉] 折扣标签不遮挡产品图', async ({ mount, page }) => {
  await mount(<ProductCard title="产品" discount={30} image="/img.jpg" />)

  await expectVisual(page, {
    rules: [
      '折扣标签完全可见，不遮挡产品图片',
      '原价和折扣价颜色区分清晰',
    ],
  })
})
```

**与 visual-test 框架的关系**：`visual-test/ct` 提供 `test`（来自 Playwright CT）和 `expectVisual` 断言。

### 6.3 L4 视觉维度：业务流程视觉连续性

**测什么**：
- 完整用户流程中每一个页面节点的视觉正确性
- 页面切换时的视觉连贯性
- 每个步骤的必要视觉元素是否存在

**与 L4 逻辑 E2E 的区别**：

| | L4 逻辑 E2E | L4 视觉 E2E |
|---|---|---|
| 问什么 | "能不能走通？" | "看起来好不好？" |
| 断言方式 | `page.click()` → `expect(url)` → `expect(data)` | `page.screenshot()` → AI 分析 |
| 能发现什么 | 数据错误、流程中断、路由错误 | 布局错乱、遮挡、Loading 缺失、颜色异常 |

**做法示例（使用 visual-test 框架）**：

```typescript
// visual-tests/e2e/checkout.visual.spec.ts
import { test, expectVisual } from 'visual-test/e2e'

test('[视觉] 下单全流程', async ({ page }) => {
  await page.goto('/')
  await expectVisual(page, { rules: ['产品列表布局整齐'] })

  await page.click('.product-card:first-child')
  await expectVisual(page, { rules: ['详情页信息完整，购买按钮可见'] })
})
```

**与 visual-test 框架的关系**：`visual-test/e2e` 提供 `test`（来自 Playwright E2E）和 `expectVisual` 断言。

### 6.4 L2 视觉与 L4 视觉的分工

| 维度 | L2 视觉测试 | L4 视觉测试 |
|------|-----------|-----------|
| 粒度 | 细粒度：一个 Card、一个 Modal、一个 Form | 粗粒度：整页、整个流程 |
| 速度 | 快（单组件渲染） | 慢（需要走完业务流程） |
| 触发时机 | 每次 PR | 预发布 smoke test / 每日定时 |
| 工具 | Playwright CT + visual-test/ct | Playwright E2E + visual-test/e2e |

### 6.5 AI 视觉断言 vs 传统视觉回归测试

| | 传统视觉回归测试 | AI 视觉测试（visual-test） |
|---|---|---|
| 判断方式 | 像素比对 | AI 理解 + 自然语言规则 |
| 脆弱性 | 极高：改 1px 就红 | 低：理解语义而非比对像素 |
| 规则表达 | 需要基准截图 | 自然语言描述 |
| 推荐度 | 不推荐 | 推荐 |

## 七、四层对照总结（含视觉维度）

| 层次 | 名称 | 逻辑断言 | 视觉断言 | Mock 策略 | 框架 |
| ---- | ---- | -------- | -------- | --------- | ---- |
| L1 | 单元测试 | 函数级：输入→输出 | 不适用 | 零 Mock | vitest |
| L2 | 组件/页面测试 | 组件级：渲染+交互 | 外观+交互体验 | Mock 全部 API | vitest + jsdom / Playwright + visual-test |
| L3 | 契约测试 | 接口级：前后端约定 | 不适用 | 真实后端路由，Mock 外部服务 | vitest（复用后端 app） |
| L4 | E2E 测试 | 用户级：业务流程 | 流程视觉连续性 | 能不 Mock 就不 Mock | Playwright + visual-test |

---

## 八、实施建议

1. **按价值排序，不要按层级排序**。先写"出 bug 会骂你的"和"重构时常改的"部分。

2. **L1 + L2 做到 80% 覆盖面就够了**。100% 组件测试覆盖是 ROI 陷阱。

3. **L3 契约测试坚持"一份测试、两种视角"**。不要在后端写一遍断言，前端再 mock 一遍。

4. **L4 E2E 只测核心流程**。登录→浏览→下单→支付。

5. **视觉测试触发策略**：L2 视觉测每次 PR 跑，L4 视觉测预发布/定时跑（控制 AI 成本）。

---

## 九、不被推荐的测试类型

- **快照测试（Snapshot Testing）**：组件换一个字就红，维护成本远超收益
- **传统视觉回归测试（像素级）**：像素比对过于脆弱。注意区别于 AI 视觉测试（visual-test 框架的做法）
- **测试测试工具本身**：不要测 `vi.fn()` 是否真的能 mock
- **100% 覆盖率追求**：95% 的 L1 + L2 覆盖率是健康的上限
